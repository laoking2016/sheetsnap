# Sheetsnap — 实施路线图

> 每一阶段都是**可独立上线的最小工作单元**（MVC — Minimum Viable Chunk）。
> 只有当前阶段验证通过后，才进入下一阶段。

---

## ~~阶段 0：项目脚手架（1–2 天）~~ ✅ 已完成

**目标**：跑通 Next.js + Supabase 基础链路，能本地开发和部署到 Vercel。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| ✅ 0.1 初始化 Next.js (App Router + TypeScript + Tailwind) | 可运行的本地项目 | `pnpm dev` 正常打开首页 |
| ✅ 0.2 配置 ESLint + Prettier | 代码风格统一 | `pnpm lint` + `pnpm type-check` 通过 |
| ✅ 0.3 创建 Supabase 项目 + 初始化数据库表 | `users`, `parse_records` 等表结构 | 建表脚本已执行，RLS 已启用 |
| ✅ 0.4 配置 NextAuth.js（Google OAuth + 邮箱） | 登录/登出按钮可用 | Google OAuth + Magic Link 均验证通过 |
| ⬜ 0.5 部署到 Vercel + 绑定自定义域名 | `app.sheetsnap.com` 可访问 | HTTPS 正常（推送后配置） |
| ✅ 0.6 设置 GitHub CI（lint + type-check） | PR 自动检查 | `.github/workflows/ci.yml` 就绪 |

**里程碑 ✅**：一个可以登录、有空数据库的空白应用已上线。

## ~~阶段 1：核心链路 — 文件上传 + 解析 → CSV 下载（3–5 天）~~ ✅ 已完成

**目标**：走通最核心的"上传 → 解析 → 输出 CSV"全流程，不处理任何边缘情况。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| ✅ 1.1 首页 UI：大按钮 "Upload Quotation File" | 简洁首页 + 拖拽上传 | 拖拽/点击均可上传 |
| ✅ 1.2 文件上传 API（`POST /api/upload`） | 接收文件，限制 10MB，类型检查 | Curl 测试通过 |
| ✅ 1.3 接入 LlamaParse | CSV 本地解析；Excel/PDF 走 LlamaParse | 返回结构化数据 |
| ✅ 1.4 列映射引擎（规则） | 从原始数据提取标准 5 列 | 关键词匹配，覆盖率良好 |
| ✅ 1.5 单位归一化规则表 | pcs→个, kg→公斤 等 12 组映射 | 无法识别的保留原文 |
| ✅ 1.6 CSV 生成 + 下载按钮 | UTF-8 BOM CSV + 在线预览前 10 行 | `file` 命令确认 BOM 正确 |

**里程碑 ✅**：用户上传一个报价单 → 点击下载 → 得到可用 CSV。**可以找内部人员手动测试了。**

## ~~阶段 2：错误处理 + 用户提示（1–2 天）~~ ✅ 已完成

**目标**：所有异常路径有明确的用户反馈，减少客服咨询。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| ✅ 2.1 非表格文件上传提示 | 前端拖拽实时拦截 + 后端校验 | 上传图片/txt 文件时内联提示 |
| ✅ 2.2 解析失败提示 | 空文件/无表格/网络错误各给不同提示 | 每个场景显示对应友好文案 |
| ✅ 2.3 单元格异常标记 | ⚠️ 图标 + 悬浮原文 + 底部警告区域 | 无法映射的列和不可解析价格标记 |
| ✅ 2.4 文件大小超限提示 | 前端 + 后端双重校验，显示实际大小 | >10MB 时阻止并提示 "Your file is X MB" |
| ✅ 2.5 全局错误边界 | ErrorBoundary 包裹应用 | 组件崩溃时不白屏，显示降级 UI |

**里程碑 ✅**：所有已知错误路径都有友好提示，可以把产品给非技术用户试用了。

## 阶段 3：预览表格 + 体验提升（1–2 天）

**目标**：解析结果在线预览，用户体验从"黑盒"变成"可见可信"。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| 3.1 在线表格预览（前 10 行） | 解析成功后自动展示 | 滚动查看、列名明确 |
| 3.2 拖拽上传支持 | `react-dropzone` 或原生拖拽 | 拖拽文件到页面可上传 |
| 3.3 加载状态动画 | 上传 → "解析中..." → 完成 | 5 秒内反馈 |
| 3.4 未登录用户引导 | 解析完成后提示 "输入邮箱接收下载链接" | 收集邮箱到数据库 |

**里程碑 ✅**：用户体验基本完整，可以小范围公开测试（邀请制）。

## 阶段 4：计费与账户系统（2–3 天）

**目标**：注册用户拥有每月免费额度，超出后可付费，收入闭环。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| 4.1 用量追踪 | 每用户/每邮箱每月解析次数 | 免费额度 20 次/月 |
| 4.2 额度不足提示 | 弹出 "已达每月免费限额" | 第 21 次上传时触发 |
| 4.3 Stripe Checkout 接入 | \$19/月（200 次）或 \$9.9/100 次（永不失效） | 支付成功后额度更新 |
| 4.4 历史记录（可选） | 用户查看已解析文件和下载记录 | 已登录用户可查 |

**里程碑 ✅**：收入闭环已跑通，可以**公开上线并开始推广**。

---

## 阶段 5：上线 & 验证（1–2 周）

**目标**：获取 50 个注册用户，验证核心指标。

| 任务 | 产出 | 验收标准 |
|------|------|---------|
| 5.1 产品 landing page 上线 | 简洁的产品介绍页 | 清晰传达价值主张 |
| 5.2 Reddit r/procurement 发帖 | 获得自然流量 | 帖子有 20+ 互动 |
| 5.3 Upwork 招募 5 个采购试用 | 每人录 1 分钟反馈视频 | 收集 5 条有效反馈 |
| 5.4 埋点分析 | 注册率 / 解析成功率 / 付费转化率 | 监控仪表盘 |
| 5.5 解析成功率追查 | 收集失败的文档，分析改进方向 | 成功率目标 ≥ 85% |

### 验证指标

| 指标 | 目标值 |
|------|--------|
| 注册用户数 | ≥ 50 |
| 解析成功率 | ≥ 85% |
| 付费转化率（免费→付费） | ≥ 3% |
| 7 天回访率 | ≥ 20% |

**里程碑 ✅**：**MVP 验证完毕 — 决定继续投资 or 放弃。**

## 阶段 6：迭代优化（验证通过后）
